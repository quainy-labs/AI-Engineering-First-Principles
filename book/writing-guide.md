# Writing Guide

## Book Standard

Write like an engineer teaching another builder.

Good chapter:

- starts from a real decision;
- explains first principles;
- names tradeoffs;
- shows where simple code beats AI;
- includes failure modes;
- requires measurement;
- ends with a proof artifact.

Bad chapter:

- lists tools;
- repeats Quainy philosophy without changing behavior;
- explains theory without system payoff;
- says "AI can help" without baseline;
- ends with vague reflection.

## Voice

Use:

- clear;
- direct;
- serious but welcoming;
- practical;
- technically honest;
- first-principles;
- no hype.

Avoid:

- "revolutionary";
- "10x";
- "AI will replace";
- "master AI fast";
- "secret prompts";
- vendor worship;
- framework worship.

## Meaningful Impact Rule

Every major section must answer:

1. What real work gets better?
2. Who benefits?
3. What was baseline?
4. What changed?
5. How do we know?
6. What can still fail?

## First-Principles Rule

For every core concept, answer:

- What is it?
- Why does it exist?
- What problem does it solve?
- What came before it?
- What tradeoff does it introduce?
- How does it fail?
- How does it fit inside a real system?

## Chapter Length Target

Target chapter length:

- 2,500 to 4,000 words for main chapters.
- 1,500 to 2,500 words for tactical chapters.
- 800 to 1,500 words for interludes.

Keep chapters short enough to finish, deep enough to change behavior.

## Required Chapter Ending

Each chapter ends with:

- decision checklist;
- failure checklist;
- proof artifact;
- active recall;
- next link.

## Project Standard

Every project needs:

- problem statement;
- user/stakeholder;
- baseline;
- inputs;
- outputs;
- system boundary;
- deterministic parts;
- AI parts;
- human parts;
- metrics;
- failure modes;
- improvement loop;
- proof artifact.

## Quainy Alignment

Principle becomes real only when learner does something differently.

Do not say:

> Build capability.

Make learner:

- build a baseline;
- ship an eval report;
- explain a tradeoff;
- reject AI where code is better;
- expose risk;
- publish proof.
